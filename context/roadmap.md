## Phase 3 ‚Äî Domain Semantics (2‚Äì4 weeks)

**Goal:** the canvas knows it's drawing clothes, not generic geometry.

Canvas content taxonomy (finalized 2026-04-24):

- **Panel** ‚Äî persistent cut shape (what gets fabric)
- **Guide** ‚Äî persistent scaffolding (points/lines/curves/arcs; handled in Phase 2 follow-ups)
- **Instrument** ‚Äî drafting aids you wield; the instrument itself never becomes pattern content. Passive (rulers, french curves, protractor) align to read or trace; active (knife) align to act on the panel (new in Phase 3)
- **Annotation** ‚Äî things that live *on* a panel: darts, notches, seam markings, labels (new in Phase 3)

### Architecture spec (locked 2026-04-27 by /plan-eng-review)

**Panel as first-class entity.** New `Document.panels: Record<PanelId, Panel>`; each Panel has a `pathId` referencing one closed `role: 'panel'` DocumentPath. DocumentPath stays focused on geometry; annotations live on Panel.

**Panel‚ÜîPath lifecycle invariant** (new R17):
- Panel exists ‚Üî a closed role='panel' path exists.
- Panel is *created* on close-by-snap or cross-panel join (the moments that flip `closed` to true).
- Panel is *destroyed* when its underlying path is deleted (cascade).
- Operations that would open a panel's path are **rejected at dispatch**, mirroring the R16 / self-intersection PREVENT-during-action pattern.

**Annotation addressing.** All annotations key by **start-node-id of the edge**, matching the existing `edgeSeamAllowances` pattern in `document.ts`. Stable across topology changes; index-based addressing breaks the moment a node is inserted mid-edge.
- Seam edge: `{ name, startNodeId, endNodeId }` ‚Äî run from start through end along the panel's `nodeIds` order
- Notch: `{ edgeStartNodeId, t }` ‚Äî t in [0, 1] along that edge
- Ease/gather: `{ fromNotchId, toNotchId, kind: 'ease' | 'gather', text? }` ‚Äî between two notches
- Dart: legs + tip as Points; mouth midpoint anchors to two leg-endpoint node IDs when legs land on the panel boundary
- Landmark label: `{ pathId, nodeId, text }` for guide landmark dots
- When a node is removed in a *legal* edit, annotations referencing it cascade-clean (silently dropped; surfaced in undo history with a descriptive label so Cmd+Z recovers them)

**Copy / mirror semantics.** Both produce new Panel + cloned Path with new node IDs. Annotations re-keyed onto the cloned IDs and carry over. Seam-PAIR links do **not** carry (the copy's seams are unpaired by default; pairing is an explicit user action). Mirror reverses node order to preserve outward-facing winding; copy gets a `" (copy)"` label suffix.

## Deferred Quality Concerns (revisit before Phase 7 / during polish pass)

Non-blocking quality issues that shouldn't ship to a second user as-is. Not full features ‚Äî fit-and-finish.

- **Seam-allowance rendering on self-intersecting panels** ‚Äî `Path.offset()` uses polygon-clipping's union, which treats a self-intersecting outline as overlapping regions and produces visually-wrong (often loopy) SA output. **No longer reachable through user action** as of 2026-04-26 (self-intersection prevention shipped at all four user-action sites: pen click, close-by-snap, node drag, handle drag). The bug remains in `Path.offset()` for any future load path (imported file, JSON paste, pre-invariant saved file), but until users have files at all this is not load-bearing. Revisit if/when persistence ships. Original noted 2026-04-24 by Justin: "the seam allowance goes crazy when you do it. I dislike the visual effect."
- **Per-edge SA rendering** ‚Äî the document model holds per-edge SA overrides already (Phase 2F's `edgeSA` helper), but `PathsLayer` currently calls `Path.offset()` with the path-level default and renders one uniform offset per panel. Per-edge variation would need either (a) a piecewise-offset algorithm that handles join geometry, or (b) explicit per-edge segment offsets joined manually. Real cost is moderate ‚Äî not blocked on anything else. Defer to polish.
- **Per-edge SA editing UI** ‚Äî when the rendering above exists, expose per-edge SA in the Inspector's segment-selected branch (input + reset-to-inherit). Trivial once rendering works.
- **Does pointer snap-to-grid belong in pattern drafting?** (flagged 2026-04-29 by Justin during the units-toggle work) ‚Äî the rebuild inherits the sketch's grid-snap behavior, but pattern drafting is fundamentally about measured construction (compass arcs, slash-and-pivot pivots, dart math) where the right point usually isn't on a grid line. Snap may be helping at the start of a draft and getting in the way once real geometry is going down. Pointer snap kept fixed at `0.5 cm` / `0.125 in` for now ‚Äî finer than the visible minor tier. Evaluation: try drafting a real recipe with snap on, then with snap off; ask whether it ever made the right thing happen vs. forced a wrong-by-one-step click. Possible outcomes: keep, remove, make a per-tool toggle, or replace with snap-to-features-only (nodes + intersections, no grid). Pairs naturally with the live-tool feedback cycle around chunks 3.8 / 3.9.
- **Cutting-line smoothness via segment-aware offset** (flagged 2026-05-04 after a failed re-fit attempt) ‚Äî `Path.offset()` runs the panel through `polygon-clipping.union` and returns the result as a polyline (lineTo-only). At print scale and on a sewn garment (the cut edge becomes the stitch line), the chord facets read as visible segmentation versus the smooth original outline. Two cheap mitigations already shipped: dropped the broken `smoothByCorners` Catmull-Rom re-fit (commit `a24d139`), bumped flattening density 64‚Üí256 (commit `313c2d6`). Acceptable but not fully smooth at high zoom. **A third mitigation ‚Äî Schneider 1990 least-squares re-fit of the offset polyline to a cubic chain ‚Äî was attempted on 2026-05-04 and abandoned.** Failure mode: `polygon-clipping` output has *exactly-collinear runs* and *one or two huge straight chords mixed in with dense smooth-curve samples within a single "run between corners"* (a 5-vertex panel with mixed cubic+line edges produced a 271-pt polyline where pt[0]‚Üípt[1]‚Üípt[2] were collinear with edge lengths 0.29 in then 8.6 in). Schneider's chord-length parameterization clusters all dense points near u=0, max-error always lands at the same pathological index, recursion always splits there, depth caps to chord facets ‚Äî the algorithm cannot converge on this distribution regardless of how the error metric or split heuristic is tweaked. **The right architectural fix is segment-aware offsetting**: in `Path.offset()` itself, traverse the *original* path edge-by-edge and produce an offset edge of the same kind ‚Äî line edges shift perpendicular by the SA distance (one line edge in, one line edge out); cubic edges go through Tiller-Hanson or de Casteljau-based cubic-offset (one cubic in, one or a few cubics out ‚Äî split where curvature exceeds offset radius). The SA path comes out as a proper cubic+line chain from the start ‚Äî no polygon-clipping, no polyline middleman, no re-fit needed. `polygon-clipping.union` stays as the fallback for the self-intersecting-input case (which is currently unreachable via user action ‚Äî see the SA-on-self-intersecting-panels entry above). Cost estimate: medium ‚Äî touches `Path.offset()`, the offset tests, and likely pushes a small new module for cubic-offset math. Not blocked on anything else. Revisit when the cutting-line facets become a real friction in Phase 6 self-use, or earlier if an external user notices.
- **Mobile / tablet layout pass for Phase 4B + Phase 4 screens** (flagged 2026-05-09 by /plan-design-review). Phase 4B ships desktop-first; below 1024px the layout reflows (wearer editor stacks diagram above grid; hub stays single-column) but isn't optimized. Mobile-optimized layouts (touch targets, condensed measurement grid, swipe-friendly session selector) belong before Phase 7's first external user. Pairs naturally with the DESIGN.md consolidation below.
- **DESIGN.md consolidation via /design-consultation** (flagged 2026-05-09 by /plan-design-review). Aesthetic decisions are currently dispersed across `context/roadmap.md` (Phase 2 follow-ups for grid + ruler, Phase 3 visual vocabulary spec for annotations + instruments + layer z-order) and Phase 4's auth pages (de-facto styling, never formalized). A formal DESIGN.md should consolidate: typography stack, font-size scale, spacing scale, color palette, button/input/modal token system, border-radius scale, focus-ring conventions, accordion conventions, modal conventions. Run before Phase 7 first-external-user.
- **Color contrast + a11y audit on Phase 4 + Phase 4B screens** (flagged 2026-05-09 by /plan-design-review). Verify 4.5:1 minimum on body text against background, 3:1 on large text. Verify all interactive elements have visible focus rings. Verify touch targets are 44px+ on the screens that will see touch use. Pairs with the mobile/tablet pass.
- **Schema-enforced relational integrity for cross-document references** (flagged 2026-05-09 by /plan-eng-review outside voice for Phase 4B). Patterns reference wearers via a weak `wearerId` in JSONB with no DB-level FK or enforced consistency, mirroring Phase 4's server-opaque-document convention. The convention will compound as more documents reference each other (parametric measurement-driven generation, future cross-document features). Revisit when a query like "all patterns for wearer X" or "all wearers above hip = Y" actually needs to run server-side. Migration path: add expression index on `(user_id, (document->>'wearerId'))` first, denormalized columns second, eventual FK + join table as a third tier. Cost is bounded; just don't paper over by piling more weak-ref conventions on top first.
- **Built-in measurement systems** (Aldrich, Armstrong, Big Four, Quiring, M√ºller & Sohn, etc.) (flagged 2026-05-13; user-CRUD tranche un-deferred + shipped 2026-05-14; built-in tranche remains here). User-defined systems now ship as part of Phase 4B v1 chunks 4B.1b + 4B.5 + NewSystemForm. The built-in tranche is what's still deferred: pre-defined drafting-system content available to every user. Quiring queued as the first Justin will collect; no timeline. Lands as additive content rows in the existing `systems` table (`is_built_in=true`, `user_id IS NULL`). Schema work + route protections already in 4B.1b carry this ‚Äî purely a content + seeding task when it ships. (Note: a starter `is_built_in` row was briefly seeded on 2026-05-14 morning and then explicitly removed in the same day's evening revision per Justin's "no system-wide template yet" call.)
- **Default tool by document state** (flagged 2026-05-01 by Justin) ‚Äî when opening a file/pattern with existing content, the starting tool should be Select (V); when the document is empty (fresh `/draft` mount or NEW after recovery-discard), the starting tool should be Pen (P). Reasoning: opening a saved pattern, the user is returning to inspect/edit existing geometry ‚Äî Select is the right starting posture. A blank page is for drawing ‚Äî Pen is the right starting posture. Currently `state.tool` is initialized once and persists across loads, which doesn't match the cognitive mode the user is in. Implementation: gate on `doc.paths` (or a more precise "has any user-drawn content" predicate) at the moments a doc gets loaded ‚Äî `documentIO.parse` success in `FileActions.handlePick`, cloud pattern open in `PatternPicker`, recovery-dialog "continue" in `RecoveryDialog`, and initial `/draft` mount; set `state.tool = paths.length ? 'select' : 'pen'`. Don't override the user's tool mid-session; only at load-edge transitions.
